# MTO ERP Change Guard

## Workflow

1. 先看最近改动上下文：
   - `git status --short`
   - `git diff --stat`
   - `git log --oneline -5`
   - 相关主文件与新拆出的分包文件
2. 再做结构性 review，先报问题再给结论。
3. 重点判断是否存在“补丁叠补丁”：
   - 同一能力出现新旧两套入口、兼容壳或残留死代码
   - 页面容器重新变胖，状态编排、渲染细节、动作逻辑再次揉回一个文件
   - 桌面端和移动端各自复制一套近似逻辑，而不是抽共享层
   - 临时样式、临时 spacing、临时判断散落多处，没有统一壳或统一约束
   - 为修一个交互继续加条件分支，而不是回到源头重整职责
4. 如果判断为补丁式实现，先重构，再继续后续开发或提交。
5. 重构时优先收成最终结构：
   - 页面入口只保留路由接线、数据获取、状态编排
   - 通用布局、toolbar、底部 action、状态徽章、筛选模型优先上收共享层
   - 删除已经失效的旧文件、兼容导出和重复实现
6. 提交前必须重新校验本轮影响范围内的类型、lint 和关键行为。

## Output Rules

- review 默认先给 Findings，再给是否需要重构的结论。
- 如果没有发现补丁式问题，要明确说明“当前结构可继续迭代”，不要空泛表扬。
- 如果发现补丁式问题，要直接实施重构，不停留在建议层。

## Commit Rules

- commit 标题使用中文 emoji 风格，格式为：`<emoji> <更新类型(范围)> <简洁中文标题>`
- commit 正文保持完整且简洁，优先写 2 到 3 段：
  - 第一段写结构或模块层面的核心改动
  - 第二段写交互、样式或行为层面的关键变化
  - 第三段只在有清理、迁移、删除旧代码时补充
- 每段内部用 `- ` 开头列关键点，避免流水账和空话
- 不写“顺手优化”“若干调整”这类低信息描述，必须指向真实模块或真实结果
