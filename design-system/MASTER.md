# MTO ERP 设计系统总则

## 1. 适用范围

本设计系统只服务于 MTO ERP 内部业务页面，不服务于营销页、品牌官网、作品展示页，也不服务于需要强叙事、重动效、3D 视觉的页面。

适用技术栈：

- React
- Vite
- Tailwind v4
- shadcn/ui

设计目标固定为：

- 信息密度优先于装饰感
- 状态表达优先于视觉炫技
- 桌面端优先提高扫描效率
- 移动端优先提高单手操作效率
- 优先保留现有业务路径、模块结构与布局壳

## 2. 全局视觉原则

### 2.1 页面气质

- 风格应稳定、克制、偏业务工具，而不是通用 SaaS landing 风格。
- 主视觉依靠版式、状态色、边框分区和字重，不依靠大面积渐变、玻璃拟态、插画装饰。
- 页面首先解决“找得到、看得懂、点得准”，再考虑美化。

### 2.2 信息层级

- 一级信息：页面标题、主状态、主金额、主单号、主操作。
- 二级信息：统计解释、辅助字段、时间、关联对象。
- 三级信息：帮助文案、限制说明、空状态提示、错误补充说明。
- 任一屏内只允许一个真正的主操作区，避免多个同权按钮竞争注意力。

## 3. 字体与字号层级

以现有 token 为准，优先使用仓库已接入字体，不新增字体体系：

- 正文：`Nunito Sans Variable`
- 特殊场景可局部使用现有等宽数字/编号样式，但不扩展新字族

字号层级建议：

- 页面标题：`text-lg` 到 `text-xl`
- 区块标题：`text-sm` 到 `text-base`
- 正文主信息：`text-sm`
- 辅助信息：`text-xs`
- 统计主值：`text-xl` 到 `text-2xl`
- 表格/列表元信息：默认 `text-xs` 或 `text-sm`

规则：

- 移动端不要为了塞更多内容降到 `text-[10px]` 常态化使用。
- 金额、数量、单号优先保持对齐和可扫描性，必要时使用等宽数字。

## 4. 颜色与状态色

只使用现有 CSS token 和 shadcn 语义色，不新增平行色板体系。

基础规则：

- 页面背景使用 `background` / `muted`
- 卡片与表单承载使用 `card`
- 边框统一来自 `border` / `input`
- 主强调使用 `primary`
- 错误使用 `destructive`

状态表达规则：

- 成功/完成：绿色系，但只用于结果性状态
- 进行中/部分完成：主色或中性强调
- 待处理：中性或浅色底
- 风险/异常：琥珀或红色，需结合文案说明风险原因

禁止：

- 为单个模块单独发明一套主题色
- 同一状态在不同页面使用相反颜色语义
- 只靠颜色表达状态，不配合标签或文本

## 5. 边框、圆角、阴影

本项目默认零圆角，必须写死：

- 设计默认值为零圆角
- 不主动把页面改成圆角产品风格
- 不引入与现有 token 冲突的新圆角体系

落地规则：

- 以现有 `--radius: 0` 为准
- 组件虽然可能仍带 `rounded-*` 类名，但视觉上不能依赖圆角建立层级
- 优先用边框、留白、底色区分层级
- 阴影只用于少量浮层、底部操作栏、悬浮按钮，不作为主结构分隔方式

## 6. 页面布局规则

### 6.1 外层布局

- 优先复用现有 `AppLayout`、Header、Sidebar、BottomNav
- 不重做全局壳层结构
- 页面内容区域以“标题区 + 筛选/操作区 + 主内容区”为基本模式

### 6.2 桌面端

- 重点提升扫描效率，支持一眼看状态、金额、数量、异常点
- 列表页优先表格或紧凑卡片，不做展示型大卡瀑布流
- 详情页优先双栏或主次分区，而不是无限向下堆叠

### 6.3 移动端

- 重点提升单手可达性和连续操作效率
- 主操作优先放到底部操作区或底部可达区域
- 搜索、筛选、状态切换应尽量靠上；提交、确认、继续下一步应尽量靠下
- 卡片信息不超过三层层级，避免长卡片

## 7. 列表页通用模式

列表页默认由以下部分组成：

1. 标题与数量摘要
2. 搜索与筛选
3. 状态分段或统计卡
4. 表格或卡片列表
5. 分页或加载更多

规则：

- 状态 Tab 要尽量带数量，帮助用户判断工作量
- 有激活筛选时必须可一键重置
- 搜索占位文案写清搜索维度
- 统计卡如果可点击，必须与筛选联动，不能只是装饰

## 8. 详情页通用模式

详情页默认由以下部分组成：

1. 顶部身份区：单号、主状态、主操作
2. 核心摘要区：关键金额、数量、进度、时间
3. 主体内容区：明细、记录、附件、历史
4. 次级操作区：导出、编辑、补充说明

规则：

- 主状态必须始终可见
- 详情页先展示结果，再展示解释
- 明细、历史、日志应分区明确，不混排
- 同类记录优先时间线、分组表格或折叠块，不建议平铺长列表

## 9. 表单、抽屉、Sheet

规则：

- 优先用现有 `shadcn/ui` 组件与项目内 `ErpSheet`
- 不为单个表单重新定义 spacing token
- 桌面端表单优先就近编辑或右侧抽屉
- 移动端表单字段高度不低于 `h-10`
- 复杂提交流程必须显示提交中状态，避免重复点击

字段规则：

- 标签、帮助文案、错误信息要同层收拢
- 必填、格式限制、文件限制要在输入前可见
- 危险操作必须二次确认，且说明后果

## 10. 底部操作栏与底部导航

移动端固定规则：

- 底部操作栏必须贴近底部导航上方，不允许悬空在内容中段
- 若底部导航已固定，业务操作栏的 sticky/fixed 逻辑必须基于真实滚动容器和 `--erp-bottom-nav-safe-h`
- 若使用 `position: sticky`，需确认其滚动容器不是被额外 spacer 人工截断
- 若页面底部已有导航占位，业务底栏通常应使用 `bottom: 0` 相对可滚动区域贴边，再通过内边距/安全区处理遮挡

交互规则：

- 主要提交按钮放右侧或更易点击位置
- 次要操作收敛进“更多”菜单，不要在底栏堆满按钮
- 底部操作栏需要边框、背景和轻阴影，保证与内容区分离

## 11. 空状态、错误状态、Loading、提交反馈

### 11.1 空状态

- 空状态必须有下一步动作，不只显示图标和一句话
- 优先给“立即创建”“上传图纸”“去筛选”这类明确入口
- 空状态文案应说明为何为空，而不是泛泛而谈

### 11.2 错误状态

- 页面内必须有错误承载区，不能只依赖 toast
- 错误提示要写清影响范围和建议动作
- 文件上传、保存失败、读取失败必须就地提示

### 11.3 Loading

- 列表用 skeleton 或占位行
- 按钮提交中必须有 loading 态
- 二次请求刷新时优先使用轻提示，不要整页闪烁重置

### 11.4 提交反馈

- 成功后应尽量刷新当前上下文，而不是强制用户返回重进
- 对话框/抽屉关闭前要保证结果可感知
- 上传成功后应清理本地临时态，避免用户误判

## 12. 组件复用原则

- 优先复用现有 `shadcn/ui`、项目布局壳、表格、Toolbar、Sheet
- 新组件先服务明确业务模式，再考虑抽象
- 所有优化建议必须能映射到真实项目文件结构，不输出 demo 式组件树

禁止：

- 不新增与现有设计系统冲突的字体、色板、圆角体系
- 不把 ERP 页面改成通用 SaaS landing 风格
- 不为了“高级感”牺牲表格密度、状态可读性和操作直达性
